Technical Architecture Deep-Dive

Five of the most important architectural decisions behind the systems above — explained with the precision of an architect, not a pitch deck.

Naive fixed-size chunking is the main reason RAG systems fail. The right strategy depends on document type and semantic density.

For this system, I used hierarchical semantic chunking: splitting at the heading level, then paragraph level, while preserving parent context. This ensures granular retrieval with broad context.

  • Sentence-boundary chunking with overlap for narrative documents (case studies, emails)
  • Semantic boundary detection using embedding cosine similarity for unstructured text
  • Table extraction and structured chunk creation for spreadsheet-derived PDFs
  • Metadata enrichment on every chunk: source, section, date, document type
  • Hybrid BM25 + dense vector retrieval (reciprocal rank fusion) for optimal recall

This agent is not a chatbot — it's a tool-calling agent operating through a deterministic sequence of LLM decisions. It uses OpenAI's function calling with strict JSON schema enforcement.

The LLM receives lead data and decides which tools to call (e.g., enrich(), score()). Each tool has defined schemas and failure modes. A human-in-the-loop hook handles low-confidence scores.

  • Structured output enforcement using JSON Schema to prevent hallucinated actions
  • Deterministic fallback: if tool call fails 3×, alert is sent to human rep via Slack webhook
  • Full audit log: every agent decision is stored with the model output, tool used, and timestamp
  • Prompt versioning: system prompt changes are version-controlled and A/B testable
  • Cost guardrails: token budget per lead limits over-calling and controls OpenAI spend

Before AI or automation works, CRM data architecture must be clean. The first two weeks were spent rebuilding the pipeline structure.

We defined a canonical data model, automated deduplication, and strictly enforced lifecycle stage rules, providing a clean foundation.

  • Canonical contact, company, and deal schema with required vs. optional fields defined
  • Automated deduplication workflow (n8n) running nightly against email and domain
  • Lifecycle stage gating: progression rules enforced via HubSpot workflow conditions
  • Company-contact association rules with auto-linking via domain matching
  • Data health dashboard: tracks completeness score per record and by rep

A BI dashboard is only as good as its pipeline. Data flows from HubSpot via Python/Pandas into a Supabase Postgres DB, queried by Metabase.

The pipeline runs every 15 minutes via a Vercel cron job, applying delta ingestion, calculating derived metrics, and alerting on failure.

  • Delta ingestion (not full-table sync) to keep API costs and latency minimal
  • Derived metric calculation in Python (velocity, ARR run-rate, forecast) stored as materialised columns
  • Idempotent upsert logic using deal/contact ID as primary key
  • Schema versioning: pipeline migrations handled with Supabase migration files
  • Alerting: any pipeline failure triggers a Resend email + Slack webhook within 2 minutes

n8n workflows usually fail without error handling, idempotency, or observability. My architecture solves all three.

Every workflow has a dedicated error branch. Webhook triggers use HMAC. Idempotency is verified via Supabase lookup before touching CRM records.

  • Error branch on every workflow — never let a silent failure go unnoticed
  • Idempotency table in Supabase: prevents duplicate emails, double-updates, ghost deals
  • Webhook HMAC verification: validates the request came from the declared source
  • Modular sub-workflows: shared logic (enrichment, CRM update) lives in reusable called workflows
  • Rate limit handling: exponential backoff on all external API calls with max retry count
